# รายงานสรุปปัญหาและวิธีการแก้ไข (Thai Lip Reading Project)

รายงานฉบับนี้รวบรวมปัญหาทางเทคนิคทั้งหมดที่พบระหว่างการนำโมเดล `auto_avsr` (ซึ่งถูกสร้างมาสำหรับภาษาอังกฤษ) มาปรับจูน (Fine-tune) เพื่อฝึกให้อ่านปากภาษาไทย ตั้งแต่เริ่มต้นโครงสร้างสถาปัตยกรรมไปจนถึงการรันบนเซิร์ฟเวอร์คลาวด์

---

## ส่วนที่ 1: ปัญหาเชิงสถาปัตยกรรมและโครงสร้างโมเดล (ตั้งแต่เริ่มโปรเจกต์)

### 1. ปัญหาด้านภาษาและ Tokenizer (Vocabulary Mismatch)
* **ปัญหาที่พบ:** โค้ดต้นฉบับและโมเดลถูกเทรนมาด้วย SentencePiece Tokenizer ของภาษาอังกฤษ (ขนาดคำศัพท์ประมาณ 1,000 คำ) เมื่อนำมาใช้กับภาษาไทย โมเดลไม่รู้จักตัวอักษรและคำไทย ทำให้ไม่สามารถแปลงภาพปากเป็นข้อความไทยได้
* **การแก้ไข:** ทำการเทรนโมเดล SentencePiece ขึ้นมาใหม่เฉพาะสำหรับภาษาไทย (`unigram_bpe_16384.model`) โดยขยายขนาดคำศัพท์ (Vocab Size) เป็น 16,384 เพื่อให้ครอบคลุมคำและพยางค์ในภาษาไทยทั้งหมด

### 2. ปัญหาการโหลดค่าน้ำหนัก (Pretrained Weights Mismatch)
* **ปัญหาที่พบ:** เมื่อเราเปลี่ยนขนาด Vocab Size เป็น 16,384 ทำให้เลเยอร์สุดท้ายของโมเดล (CTC Linear และ Decoder Output) มีขนาดใหญ่ขึ้น เมื่อพยายามโหลดค่าน้ำหนักที่พรีเทรนจากภาษาอังกฤษ (`vsr_trlrs2lrs3vox2avsp_base.pth`) PyTorch จึงแจ้ง Error ว่าขนาด Tensor ไม่ตรงกัน (Size Mismatch) และแครช
* **การแก้ไข:** เขียนโค้ดดักจับให้โหลดเฉพาะค่าน้ำหนักส่วน "การมองเห็น (Visual Encoder)" ที่ยังใช้ร่วมกันได้ และสั่งให้โมเดลสร้างค่าน้ำหนักของส่วน "ภาษา (Decoder/CTC)" ขึ้นมาใหม่แบบสุ่ม โดยตั้งค่า `strict=False` ตอนโหลด Weights

### 3. ปัญหาใช้เวลานานและทำลายความจำเดิม (Catastrophic Forgetting)
* **ปัญหาที่พบ:** การนำโมเดลขนาด 250 ล้านพารามิเตอร์มาเทรนใหม่ทั้งหมดพร้อมกันบน Dataset ภาษาไทยที่มีขนาดเล็ก ทำให้โมเดลลืมวิธีจับโฟกัสปาก (Catastrophic Forgetting) และใช้เวลาเทรนต่อ Epoch นานมาก อีกทั้งยังกิน VRAM การ์ดจอจนเต็ม
* **การแก้ไข:** ทำการ "แช่แข็ง (Freeze)" เลเยอร์ที่เกี่ยวกับการสกัดภาพทั้งหมด (`frontend`, `proj_encoder`, `encoder`) เพื่อรักษาวิชาอ่านปากเดิมไว้ และเปิดให้โมเดลฝึกอัปเดตแค่น้ำหนักในส่วนแปลผลภาษา (CTC และ Decoder) ทำให้ใช้ VRAM น้อยลงและเทรนได้เร็วขึ้นมหาศาล (ผ่านพารามิเตอร์ `--transfer-encoder`)

## ส่วนที่ 2: ปัญหาด้าน Dataset และแพลตฟอร์ม Cloud (Lightning AI / Colab)

### 4. ปัญหาการเชื่อมโยงเส้นทางไฟล์ Dataset (Path Resolution Error)
* **ปัญหาที่พบ:** เกิด `FileNotFoundError` แจ้งว่าหาไฟล์วิดีโอไม่เจอ (เช่น `DATASET/labels/DATASET/...`) เนื่องจากโค้ดต้นฉบับของ `auto_avsr` มีการบังคับแทรกชื่อโฟลเดอร์ `"labels"` เข้าไปใน Path อัตโนมัติเวลาอ่านไฟล์ CSV
* **การแก้ไข:** เข้าไปแก้ไขโค้ดในไฟล์ `auto_avsr/datamodule/data_module.py` โดยลบคำสั่ง `os.path.join(..., "labels", ...)` ออก เพื่อบังคับให้ DataLoader ใช้เส้นทางไฟล์จาก Argument `--train-file` และ `--val-file` ตามที่เราตั้งค่าไว้โดยตรง

### 5. ปัญหา "รายชื่อผี" ในไฟล์ CSV (Ghost Files in Dataset)
* **ปัญหาที่พบ:** ไฟล์ `auto_avsr_val.csv` มีรายชื่อวิดีโอตกค้างอยู่ 74 คลิปที่ถูกกรองทิ้งไประหว่างการทำ Preprocessing (วิดีโอเสีย หรือจับใบหน้าไม่ได้) ทำให้ไม่มีไฟล์วิดีโอเหล่านี้อยู่จริงในก้อน `videos.zip` พอดึงมาเทรน โมเดลหาไฟล์ไม่เจอจึงแครช
* **การแก้ไข:** เขียนสคริปต์ Python (`clean_csv`) เพื่อแสกนไฟล์ CSV และตรวจสอบกับไฟล์วิดีโอที่มีอยู่จริง หากพบรายชื่อที่ไม่มีไฟล์ ให้ตัดบรรทัดนั้นออกจากไฟล์ CSV ทันที

### 6. ปัญหาโครงสร้างการแตกไฟล์ Zip ไม่ตรงกับ CSV
* **ปัญหาที่พบ:** ไฟล์ `videos.zip` ที่บีบอัดและอัปโหลดขึ้น Hugging Face บรรจุเพียงแค่โฟลเดอร์ `videos/` ภายใน แต่ระบบคาดหวังโครงสร้างเป็น `DATASET/thai_vsr/videos/...`
* **การแก้ไข:** อัปเดตคำสั่งในคู่มือ `colab_training_guide.md` โดยใช้แฟล็ก `-d` ในการ Unzip ให้ไปลงที่โฟลเดอร์เป้าหมายโดยตรง:
  `!unzip -q -o videos.zip -d DATASET/thai_vsr/`

### 7. ปัญหาการรันคำสั่งบน Lightning AI Studio (Environment Variables)
* **ปัญหาที่พบ:** แพลตฟอร์ม Lightning AI มีปัญหาในการจัดการตัวแปร Environment ผ่าน Bash Script ทำให้บางแพ็คเกจโหลดไม่ขึ้น และเกิด `ModuleNotFoundError` สำหรับแพ็คเกจเฉพาะทาง (เช่น `torchcodec`, `torchvision`)
* **การแก้ไข:** เปลี่ยนวิธีการรันจากการใช้คำสั่ง Bash ล้วนๆ ไปใช้สคริปต์ Python (`start_training.py`) เพื่อจัดการเซ็ตตัวแปรสภาพแวดล้อมให้เสถียรขึ้น

## ส่วนที่ 3: ปัญหาด้าน Hyperparameters และช่วงเริ่มเทรน

### 8. อาการ "โมเดลติดใบ้" (Blank Token Collapse)
* **ปัญหาที่พบ:** ในการเทรนช่วงแรก (Epoch 1-10) โมเดลทายผลออกมาเป็น "ช่องว่าง (Blank)" ทั้งหมด ค่า Loss ลดลงมาที่ระดับ ~300 แล้วหยุดนิ่ง
* **สาเหตุ:** ค่าน้ำหนักของ CTC สูงเกินไป (`--ctc-weight 0.9`) เนื่องจากเราใช้สมองส่วนมองเห็นที่เคยฝึกกับภาษาอังกฤษ การใช้ CTC จับคู่แบบเฟรมต่อเฟรมโดยตรงกับภาษาไทย ทำให้โมเดลสับสน และเลือก "ทายช่องว่าง" เพื่อหลีกเลี่ยง Loss
* **การแก้ไข:** ลดน้ำหนัก `--ctc-weight` ลงเหลือ `0.1` เพื่อย้ายอำนาจการตัดสินใจ 90% ไปให้ส่วน **Decoder (Attention Mechanism)** ซึ่งฉลาดกว่าและสามารถเดาบริบทประโยคได้ดีกว่า ส่งผลให้โมเดลเริ่มเรียนรู้และหลุดจากอาการติดใบ้

### 9. ปัญหา Learning Rate ไม่เหมาะสมกับการ Fine-tune
* **ปัญหาที่พบ:** การเรียนรู้ในยุคแรกมีการแกว่งและโมเดลปรับตัวหาจุดสมดุลได้ยาก
* **การแก้ไข:** ปรับลด Learning Rate จาก `--lr 1e-3` เป็น `--lr 1e-4` เพื่อให้เหมาะกับการ Fine-tune โมเดลจึงค่อยๆ ซึมซับการแมปภาษาไทยเข้ากับโครงสร้างประโยคได้เนียนขึ้น

## ส่วนที่ 4: ปัญหาด้านการประเมินผล (Inference) และ Automation

### 10. ปัญหาการโหลดโมเดลกลับมาใช้ (Dict vs Namespace)
* **ปัญหาที่พบ:** ระหว่างโหลดโมเดลด้วย `ModelModule.load_from_checkpoint` ระบบแจ้ง `AttributeError` เนื่องจาก `args` ที่เก็บใน checkpoint ถูกแปลงเป็น `dict` แต่ในโค้ดคาดหวังให้เป็นออบเจ็กต์ `Namespace`
* **การแก้ไข:** แก้ไขโค้ดในฟังก์ชัน `__init__` ของ `ModelModule` ใน `lightning.py` เพื่อตรวจสอบและแปลง `dict` กลับเป็น `argparse.Namespace` อัตโนมัติหากจำเป็น

### 11. ปัญหามิติของภาพ (Channel Dimension Mismatch)
* **ปัญหาที่พบ:** เมื่อนำวิดีโอเข้าทดสอบ OpenCV อ่านภาพขาวดำได้เป็น 2 มิติ (H, W) ทำให้ Tensor วิดีโอรวมมีรูปร่างเป็น (T, H, W) แต่ `VideoTransform` คาดหวังมิติ Channel ด้วย (T, C, H, W) ระบบจึงนำ T ไปตีความว่าเป็นจำนวน Channel (69 Channel) และแจ้ง `TypeError`
* **การแก้ไข:** เพิ่มการแทรกมิติ Channel ด้วย `.unsqueeze(1)` หลังจากแปลงเฟรมด้วย OpenCV ทำให้รูปร่างของวิดีโอกลายเป็น (T, 1, H, W) ตรงตามที่ Data Pipeline คาดหวังพอดี

### 12. ปัญหาตัวแปร Beam Search ไม่ถูกสร้าง (Uninitialized Attribute)
* **ปัญหาที่พบ:** โค้ดทดสอบเรียกใช้ `modelmodule.beam_search(enc_feat)` และเกิด `AttributeError` เนื่องจากตัวแปรนี้ถูกตั้งค่าให้สร้างขึ้นเฉพาะตอนที่โมเดลรันกระบวนการ `forward()` แบบเต็มวงจรเท่านั้น
* **การแก้ไข:** ทำการอิมพอร์ต `get_beam_search_decoder` แล้วสั่ง Instantiate โมดูล Beam Search ขึ้นมาใหม่ตรงๆ ในโค้ด Inference แทนการดึงจาก `modelmodule`

### 13. ปัญหาการค้นหาวิดีโอและเฉลยจากชุดทดสอบ (Dataset Path Resolution)
* **ปัญหาที่พบ:** ไฟล์ `auto_avsr_val.csv` เก็บพิกัดไฟล์แบบสัมพัทธ์ (เช่น `videos/xxx.mp4`) และเก็บเฉลยเป็นรหัส Token IDs (เช่น `66 38 52 11`) ทำให้สคริปต์ทดสอบหาวิดีโอไม่เจอและไม่มีเฉลยที่อ่านออก
* **การแก้ไข:** ปรับโค้ด Inference ให้สามารถค้นหาไฟล์ CSV แบบยืดหยุ่น และนำ `row[0]` (`thai_vsr`) มารวมกับ `row[1]` (`videos/xxx.mp4`) เพื่อประกอบร่างเป็นพิกัดที่แท้จริง พร้อมทั้งดึงรหัส Token ID จากคอลัมน์ที่ 4 มาเข้า `tokenizer.post_process()` เพื่อแปลงกลับเป็นข้อความภาษาไทยเทียบเป็นเฉลยได้อย่างสมบูรณ์

### 14. ปัญหาโมเดลพูดซ้ำซ้อนในยุคแรก (Repetition Loop in Early Epochs)
* **ปัญหาที่พบ:** ในการทดสอบ Inference ช่วง Epoch แรกสุด โมเดลคายคำว่า "ไม่ไม่ไม่ไม่" ติดลูป เนื่องจากยังไม่เรียนรู้ไวยากรณ์และค่า `--ctc-weight 0.1` ทำให้อำนาจ 90% อยู่ที่ Decoder ซึ่งยังทำงานสะเปะสะปะ
* **การแก้ไข:** เพิ่ม `--ctc-weight 0.3` เพื่อรักษาจังหวะการอ่านปากแบบ "เฟรม-ต่อ-อักษร" ให้เคร่งครัดขึ้น (30%) และลดอำนาจ Decoder ลงเล็กน้อย (70%) ทำให้โมเดลหยุดการลูปคำ และเริ่มสะกดคำได้สมเหตุสมผลขึ้น

### 15. ระบบอัพโหลดและโหลดโมเดลอัตโนมัติ (Automated Hugging Face Sync)
* **ปัญหาที่พบ:** การเทรนบน Colab หรือ Cloud มีโอกาสที่รันไทม์จะดับ และสูญเสียไฟล์ Checkpoint ทำให้ต้องเริ่มเทรนใหม่ตั้งแต่ต้นเสมอ
* **การแก้ไข:**
  1. สร้าง `HuggingFacePushCallback` ใน `train.py` เพื่ออัพโหลด Checkpoint ขึ้น Hugging Face อัตโนมัติทุกๆ 50 Epoch
  2. เขียน Python Script ดักไว้ก่อนรันเทรนใน Colab เพื่อตรวจสอบว่ามีไฟล์ `epoch_xxx.ckpt` บน Hugging Face หรือไม่ หากมีให้ดาวน์โหลดลงมาและแนบคำสั่ง `--ckpt-path` ให้กระบวนการเทรน Resume ตัวเองโดยอัตโนมัติแบบไร้รอยต่อ

## ส่วนที่ 5: การปลดล็อกสมองส่วนมองเห็น (Phase 2 Unfreeze) และการปรับแต่งขั้นสูง

### 16. ปัญหา Optimizer Mismatch เมื่อทำการ Unfreeze (Resume Failure)
* **ปัญหาที่พบ:** หลังจากเทรน Phase 1 สำเร็จ และต้องการปลดล็อกการแช่แข็ง (Unfreeze) เพื่อให้โมเดลฝึกการมองเห็นริมฝีปาก เมื่อรันคำสั่งรันต่อ ระบบเกิด Error เนื่องจากสถานะของ Optimizer (AdamW) ที่ถูกเซฟไว้ใน Checkpoint เดิม ถูกผูกติดกับจำนวนพารามิเตอร์ที่น้อยกว่า (เพราะถูกแช่แข็งไว้) พอปลดล็อกแล้วจำนวนพารามิเตอร์เพิ่มขึ้น Optimizer จึงประมวลผลขนาด Tensor ไม่ตรงกัน
* **การแก้ไข:** ไม่โหลด Checkpoint แบบ Resume ตรงๆ แต่เขียนสคริปต์สกัดเฉพาะค่าน้ำหนักเพียวๆ (State Dict) ออกมาจาก `last.ckpt` แล้วเซฟเป็น `unfreeze_base.pth` เพื่อให้ `lightning.py` นำไปโหลดเป็น Pretrained Model แทนการ Resume ทำให้ Optimizer เริ่มนับหนึ่งใหม่ด้วยพารามิเตอร์เต็มรูปแบบ

### 17. ปัญหา CUDA Out of Memory ระหว่างการสลับทดสอบโมเดล (Inference)
* **ปัญหาที่พบ:** เกิด `torch.OutOfMemoryError` เมื่อพยายามรันคำสั่งเทรนโมเดลต่อ (Resume) ทันทีหลังจากที่เพิ่งรันโค้ดสุ่มวิดีโอมาให้โมเดลทดสอบอ่านปากเสร็จใน Jupyter Notebook 
* **สาเหตุ:** แม้จะรันสคริปต์อนุมานเสร็จแล้ว แต่ Jupyter Notebook (Kernel) ยังคงจองพื้นที่ VRAM (ประมาณ 8GB+) เพื่อเก็บอ็อบเจ็กต์ของโมเดลตัวทดสอบไว้ในการ์ดจอ เมื่อสั่งรันสคริปต์เทรน (ซึ่งจะเปิด Process ใหม่) จึงไม่มีที่ว่างเหลือพอสำหรับการจอง VRAM ของกระบวนการเทรน
* **การแก้ไข:** ต้องทำการ **Restart Kernel** (หรือ Restart Session ใน Colab) เพื่อล้างหน่วยความจำบนการ์ดจอที่ค้างอยู่จากการทำ Inference ทุกครั้งก่อนจะรันคำสั่งเทรนต่อ
